# Ariadne Codex Action Loop Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Convert one explicitly approved Obsidian review into a deterministic Codex-maintained Memory artifact, Mnemosyne indexing receipt, and proof that the source note was unchanged.

**Architecture:** The Obsidian plugin performs stable capture, review, approval, and immutable queue-file creation inside the existing Sync vault. Obsidian Headless Sync transports the queue to Hearken, where the existing `mnemosyne-obsidian-runner` claims the job, launches Codex with only `System/Ariadne/Memory` writable, validates the resulting wiki files, calls `/ingest`, and writes a durable receipt. The Worker remains unchanged for this slice.

**Tech Stack:** Obsidian plugin API, TypeScript 4.1, Node.js ESM, Node built-in test runner, Codex CLI, Obsidian Headless Sync, Mnemosyne Worker `/ingest`

**Execution Paths:**

- Plugin worktree: `/home/ubuntu/Documents/Codex/2026-07-13/ariadne-obsidian-save-card-current-state/.worktrees/plugin`
- Runner worktree: `/home/ubuntu/Documents/Codex/2026-07-13/ariadne-obsidian-save-card-current-state/.worktrees/runner`
- Headless vault path after separate setup approval: `/home/ubuntu/vaults/Core`

## Global Constraints

- Core and Grimoire are local views of the same Obsidian Sync remote vault.
- Obsidian Sync is the only file synchronization transport.
- Creating, clipping, pasting, or synchronizing a note never creates executable work.
- Review requires two identical note reads separated by exactly 1,000 milliseconds.
- Approval requires the current source hash to match the reviewed source hash.
- Codex may write only within `System/Ariadne/Memory`.
- The runner, not Codex, owns claims, operational logs, receipts, and Mnemosyne HTTP calls.
- The first slice supports only `operation: incorporate_note` and `allowed_domains: [knowledge]`.
- Local attachment metadata is recorded; attachment bytes are not copied, analyzed, or ingested.
- Secrets never enter the vault, prompts, logs, reports, or Git.
- Do not deploy, alter secrets, push to `main`, or merge during implementation.
